# 健康助手 Next Phase 路线图

## 1. 当前判断：登录不是“本地不能做”，而是入口还不够显眼

当前项目已经接入了 **Supabase Email Magic Link 登录**，本地和线上都可以工作；前提是：

1. `.env` 中配置了 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY`
2. Supabase 控制台已开启 Email Provider，并配置好 localhost / Vercel 的 redirect URL

现在的问题不在“能不能登录”，而在于：

- 登录入口还藏在 **健康空间** 里，不够直观
- 没有做成明显的 **登录 / 游客模式** 双入口
- 登录完成后的状态反馈、账号价值感、跨设备同步提示还不够强

下一轮应优先把登录能力做成真正用户一眼可见、可理解、可完成的产品入口。

---

## 2. 下一阶段目标

把项目从“已经可用的 AI 分诊 Beta”推进到更像真实产品的版本，重点补齐：

1. **更清晰的账号体系**
2. **更安全的药品推荐**
3. **更可信的热点地图**
4. **更丰富且更权威的疫情 / 公卫数据**
5. **更合理的个性化搜索与推荐能力**

原则上，后续所有增强都要满足：

- **先可信，再丰富**
- **先透明，再智能**
- **先医疗安全，再做推荐体验**

---

## 3. 产品原则

### 3.1 药品推荐不能像电商导购

药品模块只能先做 **OTC 非处方辅助建议 + 用药安全提醒**，不能直接给出有风险的“处方替代”式建议。  
输出重点应是：

- 适用场景
- 常见成分类型
- 不适用人群
- 与年龄 / 慢病 / 孕期 / 过敏史相关的风险提醒
- 何时应停止自行处理并转入医院

### 3.2 热点地图不能用“纯随机数”

如果用户量还不足，**不建议直接做纯随机热点**。更合理的方案是：

- 用 **定位 + 季节 + 天气 + 常见症状季节性 + 官方近期通报** 生成“趋势参考值”
- 明确标注为 **趋势估计 / 样本不足 / 官方参考**
- 当后续真实匿名上报样本够多后，再逐渐切换到真实用户信号

这样既不会显得假，也能保证页面稳定、可解释。

### 3.3 个性化推荐不应该走“广告味”

这里更适合做 **个性化检索与推荐**，而不是商业广告导向。  
推荐的对象应优先是：

- 更匹配的科室 / 医院
- 更相关的官方文章 / 指南
- 更合适的 OTC 类型提醒
- 更适合当前人群的随访建议

## 4. 核心工作流拆分

## Wave 1：登录体系产品化（优先级最高）

### 目标

让用户明显感知到“这个产品是可以登录和同步的”，而不是把登录藏在次级页面中。

### 计划

1. 把登录入口前移到：
   - 首页首屏 CTA
   - Header 导航
   - 结果页保存记录时的登录提示

2. 增加完整的账号体验：
   - 游客继续使用
   - 邮箱 Magic Link 登录
   - 登录后自动同步问诊历史和档案
   - 游客记录合并到账号空间

3. 优化状态反馈：
   - 登录成功 toast / 状态条
   - 当前账号邮箱显示
   - “本机记录 / 云端同步中 / 已同步”分层展示

### 预计涉及文件

- `src/components/Header.tsx`
- `src/components/WelcomeScreen.tsx`
- `src/components/CloudSyncCard.tsx`
- `src/lib/supabase.ts`
- `src/hooks/useHealthWorkspace.ts`

## Wave 2：药品推荐模块（OTC 安全版）

### 目标

让问诊结果不止停留在“建议就医”，而是能给出 **更实用但安全边界清晰** 的后续处理建议。

### MVP 范围

先覆盖以下高频场景：

1. 发热 / 咽痛 / 咳嗽
2. 鼻塞 / 过敏 / 感冒样症状
3. 腹泻 / 恶心 / 轻度胃肠不适
4. 轻度头痛 / 肌肉酸痛

### 输出结构

每条药品建议都应包含：

1. **推荐的是“药品类型/成分方向”而不是强推具体品牌**
2. 使用场景说明
3. 禁忌提醒
4. 老人 / 儿童 / 慢病患者特殊提醒
5. 必须就医的升级条件

### 数据来源建议

- 国家药监局（NMPA）公开资料
- 药品说明书结构化信息
- 国家基本药物目录 / 医院药学科公开科普
- WHO / 权威医学机构的常规对症处理建议

### 技术落地建议

新增：

- `src/lib/medicationCatalog.ts`
- `src/lib/medicationSafety.ts`
- `supabase/functions/medication-advisor`

数据库可增加：

- `medication_catalog`
- `medication_safety_rules`
- `medication_interaction_flags`

---

## Wave 3：更权威、更合理的热点地图

### 目标

把当前“城市趋势参考”升级成真正更可信的 **健康热点地图**，但同时保留低样本阶段的稳定体验。

### 推荐的数据分层

#### 第一层：官方公开信号

- 中国疾控中心周报 / 专题
- 国家卫健委与地方卫健委提示
- WHO / 国际背景资料
- 地方发热门诊、呼吸道就诊提醒

#### 第二层：环境与季节信号

- 定位所在城市 / 区域
- 温度、湿度、空气质量
- 季节性呼吸道 / 胃肠道高发规律

#### 第三层：匿名用户症状上报

前期用户量小的时候，不作为主信号，而作为辅助参考。  
后期用户量足够后，再逐步提高匿名真实样本的权重。

### 低样本阶段最合理的方案

先不用“完全随机”，而是用 **可复现的合成趋势模型**：

`区域基线 + 季节性 + 天气因素 + 官方通报权重 + 少量匿名上报`

并在前端明确标识：

- `官方同步`
- `趋势估计`
- `样本不足`
- `最近更新时间`

这样既真实，也不会让评委一眼觉得在“造数”。

### 技术建议

新增或改造：

- `src/lib/epidemicDataEngine.ts`
- `src/lib/hotspotScoring.ts`
- `supabase/functions/official-source-fetch`
- `supabase/functions/knowledge-sync`

数据库可增加：

- `region_health_signals`
- `official_bulletins`
- `community_symptom_reports`

---

## Wave 4：更丰富的疫情 / 公卫数据

### 目标

让公卫信息不止是一个热力图，而是形成 **“区域健康情报层”**。

### 数据内容建议

1. **近 7 天趋势**
   - 呼吸道相关
   - 发热相关
   - 胃肠道相关

2. **重点人群提醒**
   - 老年人
   - 儿童
   - 慢病患者

3. **区域型公开提示摘要**
   - 近期疾控公告
   - 本地卫健委提醒
   - 季节性健康建议

4. **透明元数据**
   - 来源
   - 同步时间
   - 置信度
   - 是否为缓存 / 兜底 / 估计值

### 同步机制建议

- 每日定时同步官方数据
- 本地缓存兜底
- 边缘函数统一清洗和结构化
- 前端展示时保留“更新时间 + 来源”标签

---

## Wave 5：个性化搜广推（建议做成“搜索 + 推荐”，弱化广告）

### 目标

根据用户画像、问诊历史、地理位置和风险等级，把信息分发做得更精准。

### 建议优先做的个性化能力

1. **个性化科室 / 医院推荐**
   - 结合定位、风险等级、开放时间、距离

2. **个性化官方内容推荐**
   - 根据症状和城市推荐更相关的权威文章

3. **个性化药品类型推荐**
   - 结合年龄、基础病、过敏史筛掉不适合的 OTC 建议

4. **个性化随访提醒**
   - 复查、观察、二次问诊、复盘趋势

### 暂不建议优先做的方向

- 商业广告位
- 非透明的黑盒推荐
- 没有证据来源的“猜你喜欢”

医疗场景里，**可信和解释性比点击率更重要**。

### 技术建议

新增表：

- `user_preferences`
- `recommendation_events`
- `saved_hospitals`
- `saved_articles`

新增逻辑：

- 用户行为埋点
- 规则 + 向量召回混合排序
- 用户反馈闭环（有帮助 / 没帮助）

---

## 5. 推荐执行顺序

### 第一阶段（最该先做）

1. 登录入口前移 + 健康空间体验升级
2. 药品推荐 MVP（OTC 安全版）
3. 热点地图从“演示感”升级到“可信趋势参考”

### 第二阶段

4. 更丰富的官方 / 公卫数据同步
5. 个性化检索与推荐

### 第三阶段

6. 数据闭环、A/B 测试、推荐优化
7. 逐步从“规则引擎”走向“更智能但可解释”的排序系统

---

## 6. 本轮计划产出

这一版路线图的结论是：

1. **登录功能本地完全能实现，不是技术障碍**
2. 现在最需要补的是 **登录入口可见性 + 用户感知**
3. **药品推荐** 应优先做成安全边界清晰的 OTC 辅助模块
4. **热点地图** 不应做纯随机，而应做“官方信号 + 环境因素 + 透明估计”的混合模型
5. **个性化搜广推** 更适合做成“个性化检索与推荐”，不要先走广告导向
